0 Big Picture First

What Problem Does This Paper Solve?

Imagine you work at a company that runs a data pipeline — an automated assembly line that takes raw data (say, taxi trip records), cleans it, enriches it, and produces a final dashboard or report. One morning the dashboard shows completely wrong numbers.

🏭
Real-world analogy Think of a car assembly line with 8 stations. Station 1 delivers raw metal → Station 2 cuts it → … → Station 8 produces the final door panel. If the door panel comes out bent, which station broke? You can't just look at the bent panel — you have to trace back through the line. That's exactly what this paper automates for data pipelines.

Modern tools can tell you what broke (the output is wrong), but not which upstream step caused it — especially when some monitoring signals are missing or incomplete. This paper:

1

Frames the problem formally

Turns "find the broken step" into a ranked list problem — rank all candidate upstream steps by how likely each is to be the root cause.

2

Builds a real benchmark (PipeRCA-Bench)

360 real incidents across 4 pipelines built from public datasets — NYC taxis, Chicago bike-share, airline flights — with injected faults and measured signals.

3

Proposes 5 ranking methods

From a simple interpretable formula (LR-H) → causal reasoning (LR-CP) → blind-spot detection (LR-BS) → machine learning (LR-L) → AI reasoning (LR-LLM).

4

Shows they work — and explains why

The best heuristic (LR-BS) identifies the root cause first on 80.8% of real incidents, and achieves perfect 100% accuracy when a specific type of monitoring failure occurs.

1 Abstract

The Summary (Abstract Decoded)

📌 What the abstract says, line by line: The abstract is the "movie trailer" of the paper — 200 words that cover the whole story. Here's what each part means.

"Modern data pipelines fail in ways that are easy to observe but difficult to diagnose."

You can see that the final output is wrong. But finding why — which of the 8 steps in the pipeline broke — requires manually tracing backwards through the whole chain. This takes hours.

"We formulate pipeline root-cause analysis as a ranked upstream-candidate retrieval problem"

Instead of just saying "something broke somewhere," the paper turns this into a ranked list: candidate step #1 is most likely the culprit, #2 next, etc. This is the same idea as Google ranking search results by relevance.

"PipeRCA-Bench: 360 labeled incidents built from four public datasets"

To test their methods, they created a reproducible test suite — 360 "fake emergencies" with known answers. Each incident is: fault injected into real data → pipeline runs → methods try to identify where the fault was injected.

The five methods: LR-H, LR-CP, LR-BS, LR-L, LR-LLM

  • LR-H — a weighted formula mixing evidence signals (like a medical diagnosis checklist)
  • LR-CP — causal propagation: follows how anomalies spread through the graph
  • LR-BS — blind-spot boosted: specifically designed for when monitoring is incomplete
  • LR-L — a machine learning model trained on historical incidents
  • LR-LLM — AI (Claude Sonnet 4.5) given structured lineage context to reason about the fault

The headline result

🎯 LR-BS achieves Top-1 accuracy of 80.8% on real pipeline data — meaning 80.8% of the time it ranks the actual root cause in position #1. Under one specific monitoring failure condition (missing root edges), it achieves perfect 100% accuracy. The learned model (LR-L) reaches 99.7%.

3 Section II — Problem Formulation

The Mathematical Definition (Made Simple)

🔑 Core idea: Turn "find what broke" into a ranked list. The paper gives this a precise mathematical form so that any method can be evaluated on the same standard.

What's a graph G = (V, E)?

🗺️
Analogy: a city road map V = the set of nodes (cities / data tables). E = directed edges (roads going one way / data flowing from one table to another). If table A feeds table B, there's an edge A → B.

Design edges vs. Runtime edges

Design edges (E_d) The connections defined in the pipeline's code or configuration. These are the planned connections — "table A should feed table B."
Runtime edges (E_r) The connections actually observed at execution time — "we saw table A was read before table B ran." Sometimes these are missing because a failing step doesn't emit any log.

Definition 1 — The RCA Task (Plain English)

Given three things:
  • (i) An observed failure — e.g., "the daily metrics table is wrong"
  • (ii) Evidence signals for every node — test pass/fail, freshness scores, anomaly indicators, recent changes
  • (iii) The full graph of what feeds what (design + runtime)
Produce: A ranked list of all upstream nodes, with the most likely root cause at position #1.

How we measure success

  • Top-1 accuracy — does the true root cause appear in slot #1? (Most demanding.)
  • Top-3 accuracy — is it in the top 3?
  • MRR (Mean Reciprocal Rank) — average of 1/(rank position). If it's always #1, MRR = 1.0. If always #2, MRR = 0.5.
  • Avg. assets inspected — how many tables does an engineer have to manually check before finding the culprit? Lower = better.
💡 Why MRR is useful

Top-1 is strict — a method either nails it first or it doesn't. MRR rewards methods that are close to correct even when they miss the top spot.

Example across 3 incidents:

Incident Root at rank 1/rank
1#11.000
2#20.500
3#40.250
MRR0.583

This matters operationally: an engineer who finds the answer at rank #2 on 100% of incidents is far better than one who finds it at rank #1 on 60% and rank #8 on the rest. Top-1 misses that distinction — MRR captures it.

💡 The operational metric matters: If LR-BS puts the root cause at position #1 on 80.8% of incidents, the average engineer only checks 0.289 tables before finding the culprit — compared to 4 tables with random guessing. That's roughly a 14× productivity improvement.

Random Walk / Diffusion

Score faults by "spreading" probabilities over a service call graph backward from the symptom. Works for continuously-monitored services, but pipeline graphs are acyclic (no loops) and have different signal types.

LLM-Based RCA

OpenRCA, Flow-of-Action, SpecRCA

Recent work using AI language models for root cause analysis. Key finding from OpenRCA: even the best LLM solves only 11.34% of failures without structured context. This paper tests if structured lineage context fixes that.

5 Section IV — The LineageRank Framework

The Five Methods Explained

Step 1: Build the Evidence Graph

For any incident, start by merging the design-time pipeline graph (from config files) with whatever runtime edges were actually captured. This gives the fused graph.

Step 2: Extract 16 Features Per Node

For each upstream candidate node, compute a 16-number "fingerprint" across four groups:

GroupFeaturesPlain English
Structural (8)proximity, blast radius, dual support, uncertainty…How close to the failure? How many downstream tables does it affect? Does it appear in both design and runtime graphs?
Observability (1)blind_spot_hintIs this node visible in the design graph but missing from runtime logs? (The "went dark" signal.)
Evidence (6)quality signal, freshness, recent change, run anomaly, contract violation, failure propagationDid data tests fail here? Is data stale? Was this recently modified? Did row counts change abnormally?
Prior (1)fault_priorBased on the type of failure observed, which node types are historically more likely to be culprits?

Now the Five Methods:

LR-H · Heuristic

The Scoring Checklist

A weighted sum of all 16 features. Like a doctor's diagnosis checklist — each symptom adds to or subtracts from the suspicion score.

score = 0.17×proximity + 0.15×blast + 0.12×blind_spot + 0.12×prior + 0.11×design_support + 0.10×freshness + … − 0.04×uncertainty

Weights were validated: the "evidence-heavy" variant gets 0.84 Top-1 vs. 0.28 for "structure-heavy" — proving evidence signals dominate.

Top-1
0.747
🔬 How the weights were determined

There's no single "correct" weight for each feature — instead the authors ran a structured sensitivity analysis across three deliberately extreme configurations, then tuned toward the dominant signal family.

Step 1 — Three probe variants

Variant How built Top-1
UniformEqual weight on every feature0.378
High-structuralStructure doubled, evidence halved0.282
High-evidenceEvidence doubled, structure halved0.838
Tuned (final)Tilted toward evidence, preserving some structure0.618

Step 2 — What the probe revealed

The 0.56-point gap between high-structural (0.282) and high-evidence (0.838) is very large. This tells you evidence features are the dominant signal — proximity and blast radius alone are almost useless. The tuned weights point toward evidence while keeping just enough structural weight to handle full-observability cases.

Step 3 — The negative weight on uncertainty (−0.04)

"Uncertainty" = a node visible in the design graph but absent from runtime. In isolation this is suspicious (the blind-spot signal), but the separate blind_spot_hint feature already captures that positively. The residual uncertainty after the blind-spot is handled correlates with nodes that are simply unobserved for innocent reasons — so the weight goes slightly negative to penalise them.

Key insight: the high-evidence variant (0.838) nearly matches LR-BS (0.808 overall), which directly motivated the LR-BS design: instead of re-tuning all weights, LR-BS applies a single multiplicative amplification on just the blind-spot signal.

LR-CP · Causal Propagation

The Gradient Detector

Root causes tend to have higher anomaly scores than their downstream descendants — the fault originated here, so signal is strongest here.

gradient(u) = max(0, anomaly(u) − avg anomaly of children)
score = 0.28×proximity + 0.30×local_ev + 0.22×gradient + 0.10×propagation + 0.10×prior

Works perfectly on schema_drift (1.000 Top-1) because the contract violation at the root is a clear gradient. Harder when signals overlap in real data.

Top-1
0.481
🔬 How the gradient coefficient (0.22) was decided

The gradient term is a derived signal — it's computed from the difference between a node's anomaly and its children's average anomaly. Because it's noisier than the raw anomaly value, it should carry less weight than local_ev.

Grid search over 3 candidates (held-out pilot incidents):

Gradient weight Top-1 Why it fails
0.10lowerGradient signal under-utilised
0.22 ✓bestBalanced: noisier than local_ev, so ~⅓ of total evidence weight
0.30lowerGradient over-weighted vs. raw evidence

Weight allocation logic:

The total evidence weight is 0.30 + 0.22 = 0.52. The gradient gets roughly one-third of that (0.22 / 0.52 ≈ 42%), reflecting it as a supporting signal, not the primary one. Propagation and prior each get 0.10 — symmetric, minimal domain knowledge. Proximity keeps 0.28 to ensure structure matters when evidence is weak.

Note: full coefficient sensitivity for LR-CP is left as future work; it is presented as a complementary directional variant rather than the primary heuristic.

LR-BS · Blind-Spot Boosted ⭐

The "Went Dark" Detector

When a step stops emitting runtime logs, it becomes invisible to monitoring — yet it's still in the design graph. This is a powerful signal it's the culprit.

base(u) = 0.35×local_ev + 0.25×proximity + 0.15×fault_prior + 0.15×failure_prop + 0.10×blast_radius
score_BS(u) = base(u) × (1 + 2.5 × blind_spot_hint(u))

When blind_spot = 1, the total multiplier is 1 + 2.5 = 3.5×. When blind_spot = 0, LR-BS reduces to the base score alone.

Top-1
0.808
🎯 Perfect 1.000 when root edges missing!
🔬 How the amplification factor (2.5) was decided

The amplification factor determines how much extra weight goes to a node that "went dark" in runtime monitoring. Too low: the blind-spot signal is drowned out. Too high: every missing node gets boosted so aggressively it degrades full-observability accuracy.

Grid search over 5 values under runtime_missing_root condition:

Factor k Effective multiplier Outcome
1.52.5×Under-amplified, misses some missing-root cases
2.03.0×Better but not perfect on missing-root
2.5 ✓3.5×Top-1 = 1.000 under missing-root. No full-obs degradation.
3.04.0×Over-amplification begins — −0.02 on full-obs
3.54.5×−0.04 full-obs degradation. Nodes innocent but "dark" over-ranked.

Why the base weights look like they do:

local_ev gets 0.35 — deliberately the highest weight, mirroring the LR-H sensitivity finding that evidence-heavy variants (0.838) drastically outperform structure-heavy ones (0.282). Under full observability (blind_spot = 0), LR-BS is essentially a 5-term evidence-plus-structure formula, and local_ev anchors it.

Under full observability, blind_spot_hint = 0 for every node, so the multiplier (1 + 2.5×0) = 1 — LR-BS reduces to its base score. This is why LR-H (with richer additive weighting) outperforms LR-BS in the full-obs condition (0.725 vs 0.642).

The Machine Learner

Train a Random Forest on historical incidents from 3 pipelines, test on the 4th (unseen). Repeats 4 times (leave-one-pipeline-out).

Top features learned: run_anomaly (0.274), recent_change (0.141), contract_violation (0.131). The model essentially learns an LR-BS-like strategy automatically.

LR-LLM · AI Reasoning

Claude Reads the Pipeline

Feed Claude Sonnet 4.5 the complete pipeline topology, per-node anomaly signals, and explicit blind-spot annotations. Ask it to reason step-by-step and assign probability scores.

final_score = 0.60×llm_prob(u) + 0.40×lineage_rank(u)

The 0.40 structural anchor prevents hallucination — zero-anomaly nodes can't float to the top just because Claude mentions them. Expected Top-1: ~0.86 (vs. 11.34% without structured context).

Top-1 (expected)
~0.86
🔬 How α = 0.60 was decided

α controls how much weight goes to the LLM's probability scores vs. the structural LR-H anchor. At α = 1.0 the LLM runs solo (hallucination risk). At α = 0 the LLM is ignored.

Grid search over 5 α values (held-out incidents):

α (LLM weight) 1−α (anchor) Outcome
0.400.60Anchor too strong — LLM reasoning dampened
0.500.50Balanced but LLM under-utilised
0.60 ✓0.40Best Top-1. LLM drives score; anchor prevents hallucination.
0.700.30Anchor weakens — hallucination starts degrading scores
0.800.20LLM fully dominates — structural signal lost

Why an anchor is necessary at all:

OpenRCA [26] showed that Claude 3.5 (the strongest LLM at the time) solved only 11.34% of real failures without structured context. Even with structured context, LLMs can still assign probability to nodes with near-zero anomaly signals if their names or types "sound plausible." The 0.40 LR-H anchor clamps those runaway scores back toward ground truth.

Key finding: LR-LLM (α=0.60) and LR-BS are statistically indistinguishable (p=0.180 after Holm-Bonferroni correction). A structured formula and a structured AI prompt achieve the same result — both benefit from lineage context injection.

The Key Insight: Observability Changes Everything

The paper tests methods under three different "monitoring conditions" — how complete is the runtime lineage?

✅

Full Observability

All runtime edges captured. LR-H wins here (Top-1: 0.725 vs. LR-BS: 0.642).

⚠️

Runtime Sparse (30%)

30% of runtime edges randomly dropped. LR-BS improves to 0.783 as more blind-spots appear.

🔕

Root Edges Missing

The root cause node emits no runtime events at all. LR-BS achieves perfect 1.000 — the "went dark" node is uniquely identified.

⚡ The practitioner takeaway: Deploy LR-BS when your monitoring coverage is known to be incomplete (e.g., sources that fail silently). Deploy LR-H when runtime lineage is reliably complete. If you have labeled historical data from 3+ structurally distinct pipelines, use LR-L for best overall performance.

✈️ What makes the BTS dual-path DAG special

In every other pipeline family, each lookup / reference table feeds exactly one downstream join node. The BTS pipeline breaks this pattern.

The real-world reason:

The BTS dataset has airport records with both location data and state labels. Two separate mart outputs need that airport reference:

  1. flights_enriched joins airport_lookup to add origin/destination names to individual flight records → feeds flights_classified → carrier_metrics & delay_metrics
  2. route_delay_metrics also directly joins airport_lookup to label origin/destination states in the aggregated route-level mart — a second, independent join that doesn't flow through the classified staging step

Why this is hard for RCA methods:

If airport_lookup is the root cause (e.g., bad_join_key fault), it corrupts two independent downstream paths at once. Both flights_enriched and route_delay_metrics break simultaneously, each showing anomaly signals. A naive proximity ranker might pick flights_enriched as the culprit (it's "closer" to some observed failures), ignoring the direct fan-out to route_delay_metrics.

Why LR-BS handles it well:

airport_lookup's blast_radius is larger than in the taxi pipelines — it has 2 outgoing paths, so more downstream nodes are affected. This amplifies the base score in LR-BS, making the source node more prominent in the ranked list. Combined with the blind-spot amplifier under missing-root conditions, LR-BS achieves its highest per-pipeline Top-1 (0.833) on BTS — better than the simpler sequential chains.

The Six Fault Types

Each fault is injected into real data via SQL statements — not simulated, but actual corruptions of the real dataset:

📝

Schema Drift

A classification column (e.g., "peak" vs. "off_peak") gets hard-coded to a wrong value. Downstream contract tests catch it.

🕰️

Stale Source

Raw data is filtered to only include rows before a cutoff date, reducing coverage. Freshness signals fire.

♊

Duplicate Ingestion

One day's data is union-ed with itself, doubling the row count. Aggregates are inflated exactly 2×.

🕳️

Missing Partition

All rows for a specific date are deleted. The output shows a gap in time series.

🌊

Null Explosion

Every 7th row has its join key set to NULL. Nulls propagate through all downstream joins, corrupting many tables. Hardest to diagnose.

🔑

Bad Join Key

The lookup table's ID column is corrupted (shifted IDs). Almost no rows match the join, producing near-empty enriched tables.

How Each Incident Is Generated

🦆
What is DuckDB?

DuckDB is an embedded analytical SQL database — think of it like SQLite, but built for fast analytical queries on large tables rather than transactional row-by-row work. It runs entirely inside a Python process with no separate server, no installation, no configuration. You just import duckdb and start running SQL.

Why does this matter here? Each incident in PipeRCA-Bench is a real pipeline execution. The tool loads actual parquet files (taxi trips, bike rides, flights) directly into DuckDB's in-memory tables, runs SQL transformations to simulate each pipeline stage, and measures how many rows come out of each stage. That row-count delta is what anchors the anomaly signals to real data behavior — not made-up numbers.

Open-source, free, runs on a laptop. No cloud, no Spark, no warehouse account needed — which is why the whole benchmark can be reproduced by anyone.

🔢 What are the 8 pipeline stages?

Every pipeline in the benchmark — regardless of domain — follows the same 3-layer node taxonomy with exactly 8 nodes. This makes cross-pipeline comparison fair.

Layer
Node (stage) — what it does — Yellow Taxi example
Source
2 nodes
raw_trips — the raw input parquet file loaded directly into DuckDB. No transformations yet — just the original dataset rows as-is.
zone_lookup — a small reference / lookup table (taxi zones, station names, airport codes). Needed for the join step. Also a source — it's read-only data, not computed.
Staging
3 nodes
valid_trips — validation stage. Filters out malformed rows: nulls in key columns, negative fares, impossible coordinates. Row count drops here if source data is bad.
trips_enriched — join / enrichment stage. Joins valid_trips with zone_lookup to add zone names to each trip. This is the join node — if the lookup table's keys are corrupted, most rows fail to match and the enriched table shrinks dramatically.
trips_classified — classification stage. Applies business logic to add computed columns: time period (peak / off-peak), fare tier (low / medium / high). Output has the same row count as enriched, but extra columns.
Mart
3 nodes
daily_zone_metrics — aggregation mart. Groups trips by date + zone, computes daily totals and averages. What a "daily dashboard" would read.
fare_band_metrics — aggregation mart. Groups by fare tier — how many low / medium / high fare trips per day.
peak_hour_metrics — aggregation mart. Groups by time period (peak vs off-peak) to produce hourly demand summaries.

Each mart reads from trips_classified. So a fault anywhere in the chain above it will cascade and distort all three mart outputs — but the fault only originated at one node. That's the RCA challenge.

1

Load real data

Open the actual parquet/CSV file in an in-memory DuckDB database — real taxi trips, real bike rides, real flights. A clean "baseline" run is also executed first, so the tool knows what normal row counts look like for this pipeline.

2

Inject fault via SQL

Run a SQL statement that corrupts one specific table — e.g., UPDATE raw_trips SET zone_id = zone_id + 99999. The root cause node is known exactly — it's whichever table the SQL touched.

3

Execute all 8 pipeline stages

Run all 8 SQL transformations in order (validate → enrich → classify → 3 marts). DuckDB returns the actual row count from each stage. A stage that "should" have 50,000 rows but now has 8,000 generates a large anomaly signal.

4

Generate evidence signals

Each node gets stochastic signals drawn from calibrated distributions anchored to the real row-count deltas: the root cause node gets high anomaly (0.48–0.84), 1–2 randomly chosen "decoy" non-root nodes get medium signals (0.34–0.71) to simulate false alarms, and unaffected nodes get low (0.04–0.22).

5

Apply observability mode

Iterations 0–4: full graph. Iterations 5–9: randomly drop 30% of runtime edges. Iterations 10–14: remove all outgoing edges from the root node entirely — simulating a step that failed silently and emitted no monitoring events.

9 Section VIII — Discussion

What Does It All Mean?

What's actually novel here?

  • 🔬

    First ranked RCA benchmark for data pipelines. There are good benchmarks for microservice RCA (RCAEval), but none for ETL/ELT pipelines. PipeRCA-Bench fills that gap with real public data and reproducible fault injection.

  • 💡

    LR-BS shows design-runtime discordance is a principled signal. "The node appears in the design graph but not in runtime" is not just a debugging clue — it's a statistically powerful discriminator that achieves perfect accuracy under the right condition.

  • 🔄

    Observability changes which method is best. There's no single best method — LR-BS wins when lineage is incomplete, LR-H wins when lineage is complete, LR-L wins when training data is available. This is a new practitioner insight.

    • Practical Guide: Which Method Should I Use?

    Decision tree for practitioners:
    • Do you have labeled historical incidents from 3+ structurally distinct pipelines? → Use LR-L
    • Is your runtime lineage coverage unreliable (sources fail silently)? → Use LR-BS
    • Is your runtime lineage reliably complete? → Use LR-H
    • Are you diagnosing null explosion faults specifically? → Avoid LR-H; use LR-BS or LR-L

    Known Limitations (Threats to Validity)

    Threat 1

    Incident Realism

    Faults are injected via SQL, not organically collected. Signals are stochastic, anchored to real row counts but not fully real production noise. The paper mitigates this with 4 real public datasets and empirically-grounded distributions.

    Threat 2

    Signal Circularity

    The signal generator was designed with knowledge of the feature space used by the methods. This creates coupling. Mitigation: PR-Adapted (zero-signal baseline) scores 0.000, confirming structural-only methods gain nothing from the design.

    Threat 3

    4-Fold CV Limit

    LR-L's cross-validation only has 4 folds (one per pipeline). Very low variance across folds (all ≥ 0.989) suggests reliability, but statistically 4 is a small number. Treat LR-L as an exploratory upper bound.

    Threat 4

    Single Root Cause Only

    The benchmark assumes one root cause per incident. Real incidents sometimes have two contributing upstream failures simultaneously. This is left for future work.

    G Glossary

    Key Terms Explained

    Data Pipeline
    An automated multi-step process that ingests raw data and transforms it into structured outputs (reports, features, dashboards) through a chain of transformations.
    Root Cause Analysis (RCA)
    The process of identifying which upstream step in a chain of operations caused an observed downstream failure.
    Lineage / Provenance
    The metadata trail recording where data came from and what transformations it went through. Design lineage = from code. Runtime lineage = from execution logs.
    DAG (Directed Acyclic Graph)
    A graph where edges have direction (A → B means A feeds B) and there are no cycles (you can't follow edges and return to your start). Data pipelines are DAGs.
    Partial Observability
    When some monitoring data is missing — e.g., a step fails silently without emitting runtime logs, creating a "blind spot."
    Blind-Spot Hint
    A binary signal (0 or 1) that is 1 when a node appears in the design graph but is absent from the runtime graph — strongly suggesting that node "went dark" during the incident.
    Top-1 Accuracy
    The fraction of incidents where the true root cause is ranked #1 by the method. Higher is better; 1.0 = always correct at the first guess.
    MRR (Mean Reciprocal Rank)
    Average of 1/(rank of true root cause). If the root is always #1, MRR = 1.0. If always #2, MRR = 0.5. Penalizes methods that rank the answer far down the list.
    Random Forest
    A machine learning method that builds many decision trees on random subsets of the data, then votes across them. Robust, interpretable, and effective on tabular data.
    Leave-One-Pipeline-Out CV
    A validation strategy: train on 3 pipelines, test on the 4th. Repeat 4 times. Tests whether the model generalizes to unseen pipeline structures rather than memorizing one.
    Bootstrap Confidence Intervals
    Re-sample the test incidents with replacement 1,500 times, compute the metric each time, take the 2.5th and 97.5th percentiles. Gives a range of likely values for the metric.
    Fault Prior
    A domain knowledge table: "if the observed fault type is stale_source, source nodes are more likely to be the root cause." Acts as a weak background probability distribution.
    DuckDB
    An embedded analytical SQL database that runs inside a Python process — no server needed. Used here to run actual ETL pipeline transformations on real public datasets.
    Blast Radius
    How many downstream tables would be affected if this node were corrupted. A node with high blast radius is a more impactful potential root cause.
    Evidence Gradient
    In LR-CP: how much higher is this node's anomaly score compared to its descendants? A positive gradient suggests the anomaly originated here (root cause), not downstream.
    Schema Drift
    A fault where a transformation outputs a column with wrong values due to a code change — e.g., a classification that was computed dynamically gets hardcoded.
    Null Explosion
    A fault where null values propagate through joins, corrupting many downstream tables — the "hardest" fault for heuristics because the anomaly signal spreads across many nodes.
